## LLM 分层架构(本票只做第一层,保留扩展)
- **LLM1 ·「听」(本票)**:监听每个终端的每条工作流/事件流,做 分析 / 归类 / 建议 / 总结。
- **LLM2+ ·(预留)**:如"验证辅助评分""对话式副驾""学习调参""跨店语义基准"等——本票不做,但接口/架构为其留位(可插拔、可加层)。

## 核心原则(护城河底线,不可破)
**LLM1 只"理解与提议",不"下裁定"。** 分析/归类/抽取的声称只写 `claimed_state`(声称)与分类标签;**验证裁定(verified/conflict)仍由确定性交叉验证(声称↔独立证据)决定**;建议仍是 advice-only、人在环。LLM 抽错/幻觉不可能造出"已验证"。低置信/歧义 → 标 pending,不自动改状态。

## LLM1 的四个功能(听到每条事件后)
1. **分析(Analyze)**:理解这条上报/事件的语义——发生了什么、涉及哪个对象/任务、声称了什么状态(含从文字/语音转写中抽取声称)。
2. **归类(Classify)**:打结构化标签——所属域(六大域之一)、任务类型(S0-7)、事件类型(打卡/上报/证据/扫码/申请支援/异常)、严重度线索、是"声称"还是"证据"还是"求助"。
3. **建议(Recommend)**:据分析产出**候选店长建议**(feed 既有 S3 编排器作候选;仍去冲突、排序、人在环批准)。
4. **总结(Summarize)**:生成摘要——如按终端/按班次/按域的动态摘要、日终总结(呼应"AI 总结"愿景)。

## 范围(第一个基本版,MVP)
- **监听**:订阅事件流(DomainEventBus/events),每条新上报/事件触发 LLM1(异步、不阻塞)。
- **可插拔适配器**:`LlmListener`/`ClaimExtractor` 接口,首实现接 DeepSeek(`base_url=https://api.deepseek.com`,`model=deepseek-chat`);provider 可换;提示词**版本化**、入日志可审计。
- **输出接回**:分析→(实体解析)→更新 Task.`claimed_state` → 触发交叉验证;归类标签写到对象/事件;候选建议进编排器;摘要经 SSE 显示在命令中心。
- **三语**:中/日/英上报均可(默认中文)。
- **可解释/多租户**:提示词、原文、输出、置信度入日志;全程 withTenant,隔离不破。
