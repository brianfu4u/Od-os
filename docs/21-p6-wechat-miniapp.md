# P6 工单 · 微信小程序员工端(产品愿景的另一半:零硬件的证据闭环)

> 状态:待开发(Maestro 拟单,创始人已 OK 推进)。负责人:工程师。审核:Maestro 逐行审。合并:创始人授权(可由 Maestro 代合并)。
> 依赖:S0–S8 + staging 已上线(Neon+Render+Vercel)。后端接口就绪:/auth/staff/wx-login、/reports、/uploads、/objects/:id/verify、SSE。参考 docs/01(§3.5 员工终端)、docs/04(小程序 API 契约)、docs/07(S1-3 上传)。

## 背景
命令中心(店长端)已上线可用。产品愿景的另一半——**员工端微信小程序**——是驱动整个交叉验证闭环的真实入口:员工在小程序里打卡、上报、扫码、拍照/语音留证,系统据此交叉验证任务是否真的完成。**零额外硬件**,靠员工日常操作 + 证据链闭环。

## 目标
做出真实的微信小程序员工端,对接**线上 staging API**,把"上报→证据→交叉验证→命令中心实时显示"这条闭环从**真实客户端**跑通。

## 范围
1. **工程结构**:新建小程序工程(建议 `packages/miniapp`,微信原生或 Taro,团队择熟),纳入 monorepo;开发用微信开发者工具(dev 关闭域名校验)对接 `https://od-os.onrender.com`。
2. **登录**:`wx.login` 拿 code → 调 `/auth/staff/wx-login` 换取会话;会话驱动租户与员工身份(生产不信任客户端自报)。开发期若无正式 AppID,用后端 dev-gated 通道跑通链路,并明确 TODO。
3. **核心页面/功能**(对应后端已支持的能力):
   - 打卡上下班(clock_in / clock_out 上报)
   - 事件/任务上报(event / task_update),幂等(clientMessageId)
   - **扫码**:扫患者就诊码(接诊/推进阶段)、扫设备/资产码(证明在用)——作为一等证据(scans)
   - **证据上传**:拍照/截图/语音 → `/uploads`,可链接到对象/上报
   - 我的任务:接收店长下发的任务、回报进度
4. **实时验证**:上报/上传后,命令中心(staging 网页)应能实时看到(SSE),并驱动交叉验证(如上传快照使 3 号房 冲突→已验证)。
5. **语言**:员工端默认中文(可保留 i18n 结构,日/英后续)。

## 验收标准(DoD)
- [ ] 在微信开发者工具模拟器里:员工登录 → 打卡 → 提交一条带**照片 + 扫码**的上报 → 该上报与证据**实时出现在 staging 命令中心**,并触发交叉验证状态变化(端到端闭环,从真实客户端)。
- [ ] 扫码作为一等证据进入验证;上传走 `/uploads` 并可关联对象。
- [ ] 幂等:重复提交同一 clientMessageId 不重复入库。
- [ ] 生产身份来自 wx 会话(openid),不信任客户端自报;开发期垫片有明确 TODO。
- [ ] 只用合成数据、不接真实 PHI。
