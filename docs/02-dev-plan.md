# OD 运营系统 · 工程开发计划与团队分工 (v1)

## 0 · 概览与铁律

**目标:** 把已定稿的「结构 + UI」建成可运行、可销售的多租户产品。

**现有资产(设计事实来源):** 三语指挥官命令中心 UI 原型;核心结构设计文档;架构可视化(对象-关系图 / 七阶段循环图 / 分层图)。

**团队:** 2–3 名开发工程师 + 产品负责人(指挥)+ AI(架构与 UI 设计、逐票规格、代码评审)。

**三条铁律:** ① 先窄后宽(MVP 只做 5 个关键任务类型);② 人在环优先(早期不做高风险自动化);③ 合成数据先行(先合成跑通,再接真实与 PHI)。

## 1 · 技术栈建议(适合小团队)

少而成熟、全栈同语言。

| 层 | 选型建议 | 理由 |
|---|---|---|
| 前端 | Next.js (React) + TypeScript + Tailwind + next-intl | 直接承接三语原型;生态成熟 |
| 后端 | Node / NestJS (TypeScript) | 与前端同语言,小团队高效 |
| 数据库 | Postgres + 行级安全(RLS,多租户)+ pgvector | 一库搞定关系+向量 |
| 事件/账本 | append-only 事件表 + 验证账本表 | 支撑 agentic loop 与审计 |
| LLM | LLM API(声称抽取 / 验证推理 / 生成建议) | 不自建模型 |
| 接入 | Webhook 接收(先 1 个 IM)+ 文件上传 | 最小可用 |
| 实时 | WebSocket / SSE | 命令中心实时刷新 |
| 基础设施 | 前端 Vercel + 后端容器 + 托管 Postgres + GitHub Actions CI | 轻 DevOps |

> 若团队更熟 Python:可将「抽取 + 交叉验证」拆为 FastAPI 独立服务,其余保持 TypeScript。

## 2 · 工作流拆分(7 个 workstream)

| # | Workstream | 职责 |
|---|---|---|
| A | 本体与数据核心 | 对象/关系/状态三元组、对象库、事件账本、多租户脚手架 |
| B | 接入与信号映射 | IM/上传连接器 + LLM 声称抽取 + 实体解析 |
| C | 交叉验证引擎 | 证据模型、置信度评分、验证状态、触发规则、账本写入 |
| D | 智能体与编排 | 域 agent、预测/发现函数、建议生成、orchestrator 排序 |
| E | 动作回写与审批 | 动作类型、审批闸门、ActionLog、人在环 |
| F | 呈现层 | 命令中心 + 角色视图 + 下钻页 + 实时 + 三语 |
| G | 横切 | Auth/RBAC、多租户隔离、DevOps/CI、安全/PHI、测试数据 |

## 3 · 工程师分工

**3 人版(推荐)——纵向拥有 + 共享核心:**

| 工程师 | 角色 | 主拥有 |
|---|---|---|
| E1 | 平台与数据(偏后端) | A 本体数据核心 + G 横切 —— 地基,最先动 |
| E2 | 智能与验证(偏后端/AI) | B 接入+抽取 + C 交叉验证 + D 函数/agent —— 系统大脑 |
| E3 | 体验与前端 | F 命令中心/角色视图/下钻 + E 审批交互 —— 把原型变真应用 |

**2 人版:** 工程师A 全栈后端(A+B+C+D+E+G 核心);工程师B 全栈前端+集成(F + E-UI + 接入 + QA)。

**责任矩阵(RACI · R=主做 A=审定 C=协作 S=支持):**

| Workstream | E1 | E2 | E3 |
|---|---|---|---|
| A 本体数据 | R | C | – |
| B 接入抽取 | C | R | – |
| C 交叉验证 | C | R | – |
| D 智能编排 | C | R | C |
| E 动作审批 | R | C | R |
| F 呈现 | – | – | R |
| G 横切 | R | C | C |

## 4 · 里程碑与排期

**假设:** 2 周一个 sprint;3 名中级工程师;约 7 个月到「可销售」。Phase 1 MVP 约第 12 周。

| 阶段 | 周 | 目标 | 里程碑 |
|---|---|---|---|
| Phase 1 · MVP | W1–12 | 单店「只读 + 建议」闭环 | M1 可用 MVP |
| Phase 2 · 全功能 | W13–20 | 动作回写 + 六域全 + 学习环 | M2 单店全功能 |
| Phase 3 · 产品化 | W21–28 | 多租户 + 第二家试点 | M3 可销售产品 |

**Sprint 明细:**

| Sprint | 周 | 焦点 | 主责 |
|---|---|---|---|
| S0 | 1–2 | 地基:仓/CI/部署 + 本体核心 schema + 原型入壳 + 冻结 5 个任务类型 | 全员(E1 主) |
| S1 | 3–4 | 接入+抽取:1 个 IM 连接器+上传;LLM 抽取→Task.claimed;对象 API | E2/E1 |

## 5 · 关键路径与依赖

**关键路径 = A 本体核心。** 必须最先落地。

```
A 本体核心(S0)
  └→ B 接入抽取(依赖对象模型)
        └→ C 交叉验证(依赖声称+对象)
              └→ D 推理/建议 └→ E 动作回写
F 呈现层:用「现有原型」作为 API 契约,对 mock 数据并行开发,不等后端。
```

## 6 · 协作方式

2 周 sprint,明确 DoD;每周向指挥演示;每票带规格与验收标准;先合成后真实;锁定 MVP 范围;每个 PR 评审后合并。

## 7 · 风险与对策

| 风险 | 对策 |
|---|---|
| LLM 抽取/验证不准 | 窄任务类型、必需证据门槛、低置信转人工、账本回流优化 |
| 员工不愿在 IM 记录 | 用已在用的 IM、零/低摩擦、上报做成一键+拍照 |
| 范围蔓延 | 锁定 5 个任务类型,新需求进 backlog |
| PHI/合规 | 先合成数据;上真实 PHI 前做合规 pass;最小化+RBAC+审计账本 |
| 小团队/经验不足 | AI 产出逐票规格+脚手架+评审;每周演示纠偏 |
