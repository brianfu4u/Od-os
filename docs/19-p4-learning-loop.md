# P4 工单 · 学习环(S8)——让系统越用越准(确定性、可解释、人在环)

> 状态:待开发(Maestro 拟单,创始人已 OK 推进)。负责人:工程师。审核:Maestro 逐行审。合并:创始人授权(可由 Maestro 代合并)。
> 依赖:P0–P3 已合并进 main(感知→验证→建议→批准→执行→撤销 + 命令中心接真数据)。基于 docs/08(S2 交叉验证)、docs/09(S3 建议/编排)、docs/11(S0-7 配置)、docs/13(路线图)。

## 背景
现在系统能验证、能建议、能执行。但它还"不会长记性":同样的误判、被反复忽略的建议、被撤销的动作,不会反馈回去改进。P4 加一个**确定性、可解释**的学习闭环,让证据权重、置信度门槛、建议排序随真实使用反馈而调整——这正是护城河"越用越准"的兑现。**不做黑箱**:每次调整都有依据、有记录、可回滚。

## 目标
把「结果反馈」沉淀下来,并据此**有界地、可解释地**调整每任务类型的证据权重/门槛与建议排序;人在环——学习产出的是"建议的参数调整 + 依据",而非静默改模型。

## 范围
1. **反馈捕获(追加式)**:把这些信号落成不可篡改的 `learning_feedback`(append-only,和 events/账本同口径:forbid_mutation 触发器 + 只 SELECT/INSERT + RLS FORCE + 租户策略):
   - 建议被 **批准 / 忽略 / 撤销**(来自 P2/P3);
   - 验证裁定后被**人工更正**(如"冲突"实际没问题、或"已验证"其实没做)——需要一个轻量的"更正"入口/事件。
2. **确定性学习批处理**:一个可重复运行的 `learn` 过程(按租户),消费 `learning_feedback`,产出**有界调整**:
   - 每任务类型的 `evidenceWeights` / `confidenceThreshold`(S0-7 配置)按反馈方向**小步、有上下限地**微调(如某类证据总与真实完成正相关→略升其权重;反之略降);
   - 建议排序:某域/某动作类型的建议**长期被忽略**→在编排器里下调其优先级。
   - 每次 `learn` 运行写一条**审计记录**(调整了什么、依据哪些反馈、前后值),可回滚到上一版参数。
3. **接回主链**:调整后的参数被 S2 打分与 S3 编排读取(延续 docs/08/09 的"配置可被覆盖"机制),形成闭环。
4. **边界**:调整必须**有界**(设 min/max 与单次步长上限),避免被少量异常反馈带偏;低样本量时不调整或仅弱调整。LLM 评分器仍是可插拔 seam,本票不引入。

## 验收标准(DoD)
- [ ] `learning_feedback` 落库为 append-only(改/删被数据库层挡),租户隔离,涵盖批准/忽略/撤销/人工更正四类信号。
- [ ] `learn` 批处理可重复运行,产出每任务类型的参数调整 + 一条可读的审计记录(依据+前后值),且调整在设定的 min/max/步长上限内。
- [ ] 反馈驱动排序:构造"某建议类型被反复忽略"的合成数据 → 该类型建议优先级下降(有测试断言)。
- [ ] 反馈驱动权重:构造"某证据类型反复与真实完成正相关"→ 其权重按预期方向变化(有测试断言,且不越界)。
- [ ] 可回滚到上一版参数;低样本量时不过度调整(有测试)。
- [ ] 跨租户隔离仍全绿;S2/S3 读到调整后的参数形成闭环;人在环(学习只调参数、可审计,不静默执行动作)。

## 不做(避免镀金 / 留后续)
LLM 评分器/生成式再排序;跨店匿名基准(需多租户聚合与合规,留产品化阶段);真实外部集成;上线部署(需创始人基础设施凭证,单独立项)。
