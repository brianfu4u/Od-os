# P2 工单 · S4 动作回写层(让「批准」真正干活,人在环 + 审批闸门)

> 状态:待开发(Maestro 拟单,创始人已 OK 推进)。负责人:工程师。审核:Maestro 逐行审。合并:创始人授权(可由 Maestro 代合并)。
> 依赖:P1(S0-3 会话鉴权)已合并进 main。基于 docs/09(S3 建议 + proposedActions)、docs/13(路线图)、docs/01(动作/写回)。

## 背景
现状(S3):店长「批准/忽略/稍后」一条建议时,系统**只记录意图 + 写事件**,不执行任何现实动作(见 `recommendation.repository.ts` 的 `setStatus`,注释 “no world action in S3”)。P2 让**被批准的低风险动作真正执行**——但严守边界:只做**系统内部的本体写回**,不触碰不可逆的现实世界副作用;高风险动作仍只提议、由人手动执行。

## 目标
把「批准」接到一个**动作回写层**:批准一条建议 → 若其动作在**低风险可执行白名单**内 → 在 `withTenant()` 事务内执行对应的**本体写回** + 追加一条不可变 `ActionLog` + 发事件;否则 → 维持现状(仅记录意图,等人工执行)。全程人在环、可审计、尽量可回滚。

## 范围
1. **动作类型注册表 + 低风险白名单**。MVP 先做这几个**纯内部写回**(不发外部消息/不下真实订单/不动支付):
   - `inventory_reorder` → 创建一个补货 Task / 采购意向对象(内部)。
   - `reassign_task` → 改某 Task 的 `assignedTo`(内部)。
   - `equipment_offline` → 置 Equipment `status=offline` 并创建一条校准 Task(内部)。
   - `flag_review_followup` → 为差评创建一条跟进 Task(内部)。
   **高风险(默认不在白名单,永不自动执行)**:真实给患者发消息、提交理赔、调用供应商/支付等外部副作用——仍只提议,人工手动执行。
2. **执行管线**:`approve` → 查动作类型 → 在白名单内则调用对应 handler 执行本体写回(全部走 `withTenant()`),否则仅记录意图(现状行为)。
3. **ActionLog(追加式)**:记录 `{recommendationId, actionType, params, actor, result, before/after 或可回滚引用, at}`;像 `events`/`verification_ledger` 一样**只可 INSERT**(触发器 + 授权层双保险)。
4. **幂等**:重复批准同一建议不重复执行(用状态/幂等键防重)。
5. **可回滚**:可逆动作记录足够的前态以支持撤销(至少 `reassign_task`、`equipment_offline` 提供 undo 路径)。

## 验收标准(DoD)
- [ ] 批准一个**低风险**动作 → 目标本体对象**真的发生内部状态变化** + 写入一条 `ActionLog` + 发事件,全部在 `withTenant()` 内。
- [ ] 批准一个**高风险**动作 → **不执行**,仅记录意图(白名单闸门生效)。
- [ ] **幂等**:对同一建议重复批准,不产生第二次执行/第二条重复写回。
- [ ] **可回滚**:对可逆动作能撤销并恢复前态(带测试)。
- [ ] 跨租户隔离测试仍全绿;`events`/`verification_ledger`/`ActionLog` 仍为 append-only;`withTenant()` 仍是唯一业务数据通道。
- [ ] 全程人在环:只有经理批准才执行;高风险永不自动执行。

## 不做(避免镀金 / 留后续)
真实外部世界集成(供应商下单 API、理赔提交、患者短信/邮件/微信推送、支付)——需真实凭证与合规,留后续单独立项;学习环(S8);LLM 动作措辞/再排序。
